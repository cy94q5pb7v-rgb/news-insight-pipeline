# Генерация инсайтов из новостей

Как из новостей (и загруженных материалов) формируются инсайты/гипотезы, как
устроена очередь фоновых задач, модель данных гипотез и их отображение.

> Терминология. *Гипотеза/инсайт* — структурированное наблюдение (что произошло
> + что это значит + что с этим делать), выведенное LLM-агентом. В коде сущность
> называется `hypotheses`, но по смыслу это «инсайты». *Фоновая задача (job)* —
> единица работы в очереди оркестратора. *Жизненный цикл (lifecycle)* — стадии,
> через которые проходит гипотеза от появления до архивации.

---

## 1. orchestrator.py — модель фоновых задач

LLM-генерация инсайтов долгая (минуты) и ресурсоёмкая, поэтому запускается не
синхронно в HTTP-запросе, а как фоновая задача в очереди. `orchestrator.py` —
самодостаточный асинхронный планировщик на SQLite.

**Хранилище:** таблицы `jobs` и `job_events` (создаются `ensure_schema`). Каждая
задача — строка со статусом (`queued/running/done/error/cancelled/interrupted`),
тарифом, приоритетом, payload и результатом (JSON).

**Регистрация обработчиков:** `register_kind(name, handler, *, timeout_s, serial,
recover, max_retries)`. Обработчик — функция `handler(user, payload, job_id) ->
dict`. Может быть синхронной (выполнится в `asyncio.to_thread`) или асинхронной.
Генерация инсайтов регистрируется как:

```python
orchestrator.register_kind(
    "insights_regen", _run_insights_job,
    timeout_s=960, serial=True, recover=False
)
```

`serial=True` — только один прогон инсайтов одновременно (через per-kind
`asyncio.Lock`). `recover=False` — после рестарта сервера незавершённый прогон не
переочередуется (он не идемпотентен).

**Приоритеты и лимиты:**
- Приоритет = `tier_rank * 10^13 + submitted_ms` (`_priority_score`) — сначала по
  тарифу (admin > premium > basic), внутри тарифа — FIFO по времени подачи.
- Семафоры: глобальный (`GLOBAL_MAX`, по умолчанию 3), per-user (слоты по тарифу
  `TIER_SLOTS`), per-kind lock для serial-задач.
- Очередь — min-heap (`heapq`) по приоритету, синхронизированный с БД.

**Диспетчер** (`_dispatcher_loop`): ждёт события `_wake`, достаёт задачу из heap,
берёт глобальный семафор, запускает `_run_job` как отдельную asyncio-таску.
`_run_job` берёт per-user и per-kind семафоры, переводит задачу в `running`,
запускает обработчик с таймаутом (`asyncio.wait_for`), пишет результат/ошибку.

**Ретраи:** обработчик может бросить `TransientError` (сетевой сбой, таймаут
агента) — оркестратор переочередит задачу с экспоненциальной задержкой
(`_retry_delay_s`: 30, 90, 270, 810с) до `max_retries`. Реализовано через сдвиг
приоритета в будущее (`_schedule_retry`) + таймер-будильник.

**Восстановление** (`_recover_interrupted`): на старте задачи в статусе `running`
(остались от упавшего процесса) либо переочередуются (если `recover=True` и
попыток <2), либо помечаются `interrupted`.

**Отмена** (`cancel`): ставит флаг `cancel_req`; если задача уже бежит — шлёт
`terminate()` отслеживаемому subprocess (через `track_process`). Обработчик может
проверять `is_cancelled(job_id)`.

**Прочее:** `set_phase` — тонкий статус для polling UI («agent», «storing»);
`metrics` — p50/p95 длительности и ожидания, error-rate за окно; `list_for_admin`
— снимок очереди; `_daily_cleanup_loop` — удаление задач старше 30 дней.

## 2. routes_insights.py — логика генерации

Основная функция-обработчик — `_run_insights_job(user, payload, job_id)`.

### 2a. Сбор источников — `_collect_new_docs_for_insights()`

Объединяет два типа источников в единый список «документов» для анализа:

- **Материалы** из `kb_docs` (только `moderation_status='approved'`), последние
  `INSIGHTS_MAX_DOCS` (60), обрезанные до `INSIGHTS_DOC_CHARS` (2400) символов
  каждый. Помечаются `kind="material"`.
- **Новости** из travel- и packages-архивов (`_collect_news_for_insights`),
  отсортированные по дате. Помечаются `kind="news"`, id вида `news:<hash>`.
  Перед использованием новости пишутся снимком в `kb_news_items`
  (`_upsert_news_snapshot`), чтобы гипотезы могли на них ссылаться.

**Инкрементальность:** уже обработанные документы отслеживаются в таблице
`kb_insight_docs`. При первом прогоне (`is_first`) берутся все; при последующих —
только новые (которых нет в `kb_insight_docs`). После прогона новые id помечаются
обработанными (`_mark_docs_processed`).

### 2b. Промпт и вызов LLM

`_build_insights_prompt(docs)` склеивает большой системный промпт `INSIGHTS_PROMPT`
(роль «продуктовый аналитик премиального блока банка», исследовательская рамка,
строгое различение «факт ≠ инсайт», 5 обязательных полей инсайта, правила перевода
англицизмов, примеры хороших/плохих инсайтов) с телом каждого документа
(заголовок + источник + теги + контент). Общий лимит `INSIGHTS_PROMPT_MAX`
(180 000 символов).

Вызов агента — `subprocess.Popen([OPENCLAW, "agent", "--session-id", ..., "--message",
prompt, "--json"])` с таймаутом 900с. Фаза задачи переключается в «agent».
Процесс регистрируется для возможной отмены. Ответ парсится `_extract_reply` (из
`core/llm_agent.py`) + `_parse_insights_json` (балансировка скобок, находит первый
валидный JSON-объект).

### 2c. Валидация и запись — `_validate_and_store()`

Для каждой гипотезы из ответа агента:

- Категория нормализуется (`_normalize_category`) к одной из
  `("ПУ/подписки", "Travel", "UX/UI")`; без валидной категории — skip.
- `confidence` зажимается в [0, 1]. `validated = 1` если `confidence >
  INSIGHTS_MIN_CONF` (0.9).
- Источники (`sources`) фильтруются: `doc_id` должен быть в наборе валидных id
  текущего прогона. Без statement или без источников — гипотеза отбрасывается.
- `source_kind` выводится из источников: все `news:` → "news", ни одного → "material",
  смесь → "mixed".
- **Dedup-защита жизненного цикла:** перед удалением старых гипотез читаются
  «защищённые» (со статусом `in_review/validated/adopted`). Если новая гипотеза в
  той же категории имеет Jaccard-сходство statement ≥ `LIFECYCLE_DEDUP_THRESHOLD`
  (0.7) с защищённой — её источники **дописываются** в защищённую (сохраняя
  владельца и статус), а дубликат не создаётся.
- При полном (не `append`) прогоне старые гипотезы (кроме защищённых) удаляются.

Запись — в `kb_hypotheses` + `kb_hypothesis_sources`. Прогон логируется в
`kb_insight_runs` (статус, число документов/гипотез/валидированных, ошибка).

## 3. Жизненный цикл гипотез

Гипотеза проходит стадии (`LIFECYCLE_STATUSES`):

```
synthesized → in_review → validated → adopted → archived
     ▲            │ │           │          │        │
     └────────────┘ └───────────┴──────────┴────────┘ (archived → synthesized с confirm)
```

Разрешённые переходы заданы в `ALLOWED_TRANSITIONS` (через UI). Особенности по
коду:
- `synthesized → validated` напрямую нельзя — только через `in_review`.
- Переход в `in_review` авто-назначает владельца (`owner_username = user`), если
  его не было.
- Переход в `validated/adopted` требует владельца (иначе 409 `assign_owner_first`).
- Архивация требует `confirm:true`.
- `validated → in_review` намеренно отсутствует в UI (только админ через SQL).

API жизненного цикла (`routes_insights.py`):
- `POST /kb/insights/{id}/lifecycle` — смена статуса.
- `POST /kb/insights/{id}/owner` — назначение владельца (не-админ может только
  «забрать» себе ничейную или отпустить свою; атомарный claim).
- `POST /kb/insights/{id}/next_check` — дата следующей проверки (для «просроченных»
  гипотез, `is_overdue`).
- `GET /kb/insights/mine`, `/badge` — «мои» гипотезы и счётчик (in_review +
  overdue) для бейджа в UI.

Признак «новизны» (`_is_new_for_run`): гипотеза «новая», если создана сегодня
(UTC) И произведена самым свежим завершённым успешным прогоном — следующий
успешный прогон автоматически снимает флаг со старых.

## 4. Модель данных гипотез/инсайтов

(Полностью — в `docs/data-model.md`; здесь кратко связи.)

- **`kb_hypotheses`** — сама гипотеза: `statement` (наблюдение+интерпретация),
  `rationale` (импликация+обоснование), `category`, `confidence`, `validated`,
  `evidence_count`, `source_kind`, поля жизненного цикла (`lifecycle_status`,
  `owner_username`, `next_check_at`), `run_id`.
- **`kb_hypothesis_sources`** — связь гипотеза↔источник (M:N):
  `(hypothesis_id, doc_id, excerpt, is_origin)`. `doc_id` либо id материала
  (`kb_docs`), либо `news:<hash>` (`kb_news_items`).
- **`kb_insight_runs`** — журнал прогонов генерации.
- **`kb_insight_docs`** — какие документы уже обработаны (инкрементальность).
- **`kb_news_items`** — снимок новостей-источников.

В API детали гипотезы (`GET /kb/insights/{id}`) источники резолвятся: материалы
join'ятся с `kb_docs` (+оценка качества `_score_doc_quality`), новости — с
`kb_news_items`. Граф связей между гипотезами (`edges` в `/kb/insights/data`)
строится по общим источникам.

## 5. Отображение на портале

`insights/template_insights.py` — одна большая строка-константа `INSIGHTS_HTML`
(HTML+CSS+JS, ~1.9k строк). Это самодостаточная SPA-страница (светлая/тёмная
тема, sticky-фильтры, переключатель «AS IS / Архив», поиск, drawer с деталями
гипотезы и её источниками, граф связей). Логики на сервере в этом файле нет —
только разметка с плейсхолдерами (`__USER__`, `__CANUPLOAD__`, `__CANMOD__`,
`__INIT__`, `__ADMIN_ITEM__`, `__FEEDBACK_WIDGET__`), которые роутер
(`kb_insights_page`) подставляет перед отдачей. Данные страница тянет AJAX'ом из
`/kb/insights/data`, `/kb/insights/section/{id}`, `/kb/insights/{id}` и пр.

> Замечание: `template_insights.py` упоминает интеграцию с Telegram Web App
> (`telegram-web-app.js`) — страница рассчитана работать и как мини-приложение в
> Telegram, и как обычная веб-страница.
